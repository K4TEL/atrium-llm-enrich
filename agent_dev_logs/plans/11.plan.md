## Context

Issue #11 asks which **input format** should feed the LLMs, grounded in the formats that existing
Document-Understanding benchmarks actually use, toward a FAIR collection. The issue opens with a
format × detail-level matrix (PDF · InDesign XML · HTML+CSS · DOCX · PAGE/ALTO XML · MD · TXT) and
two leads: route page images via
[`ufal/atrium-page-classification`](https://github.com/ufal/atrium-page-classification), and check
[Grobid](https://grobid.org/).

Three comments (K4TEL) sharpened the question into a near-decision:
- **2026-07-20** — "Why Markdown wins": HTML = token bloat, TXT = structure loss, MD = the
  token-efficiency/structure sweet spot; complex layouts need modern parsers (Docling/Marker; OTSL
  tables ≈80 % fewer tokens than HTML; LaTeX for math); **critical requirement: inject
  `<!-- Page N -->` markers** so page-level citations survive the conversion. PoC recommendation:
  Docling or Marker on our hardest PDFs. (Full Gemini 3.1 Pro DR report attached to the issue:
  `Roadmap-LLM-Document-Retrieval.md`.)
- **2026-07-21 07:14** — DR summary (`research-issue-11.md` attached): (1) MD is the optimal lingua
  franca; (2) spatial provenance via embedded bounding markers replicates PAGE/ALTO's advantage
  inside MD; (3) tooling: Docling/Marker over legacy heuristic parsers, Grobid solid but older;
  (4) FAIR alignment via hierarchical metadata "roadmaps" (`SKILL.md`/`INDEX.md`) navigated by an
  agent instead of chunk-RAG.
- **2026-07-21 07:37** — cross-post of the visual-layout-cues idea that became issue #10's
  implementation.

**Relation to the rest of the repo.** Issue [#10](https://github.com/ufal/atrium-llm-enrich/issues/10)
already **implemented** the mechanism this issue converges on: `api_util/doc_to_visual_md.py`
(+ `layout_md/docx_to_md/pdf_to_md`, landed `456ebb2`) emits page-sectioned Markdown with
`<!-- PAGE_BREAK: pg_N -->`, `<!-- BBOX -->`, `<!-- FONT -->`, … cues — i.e. the "annotated
Markdown" of the DR conclusion, with a richer cue set than bare `<!-- Page N -->`. Hub
[ufal/atrium-project#22](https://github.com/ufal/atrium-project/issues/22) owns the **evaluation**
harness (`sample_stratify.py` + `eval_metrics.py` + `bench_compare.py`) that must arbitrate the
remaining tool/format choices empirically. What #11 adds on top of #10 is the **decision layer**:
which formats are inputs, which are intermediates, and which single format is the LLM diet.

## §1 — Format decision matrix (verdicts)

Extending the issue's matrix with token economics, spatial provenance, and repo support today:

| Format | Detail available | LLM diet? | Verdict for this repo |
|---|---|---|---|
| **PDF** | Max visual + textual | ❌ | **Source format, never the diet.** Route via #10: digital-born → extract; scans/curves → OCR (pending). |
| **InDesign XML (IDML)** | Fonts, spacing, graphics regions, complex layout | ❌ | **Skip.** Authoring-side format; AMCR/archive inputs don't ship IDML. Revisit only if native sources appear. |
| **HTML+CSS** | Graphics + formatted text + complex layouts | ❌ | **Intermediate only** (e.g. mammoth's semantic HTML from DOCX). As a diet it token-bloats (`<div>`/`<span>`/styles) — the DR's central negative finding. |
| **DOCX** | Same as HTML | ❌ | **Input format** (real Unicode layer) — handled by `docx_to_md.py`; not a diet. |
| **PAGE / ALTO XML** | Text regions, word-level positions, graphics regions, hyphenation — no visuals | ⚠️ | **Spatial source of truth**, not a diet: verbose XML, but it feeds the **line-level** entry (TEITOK/CSV) and `xml_to_md.py`; OCR outputs (PERO PAGE-XML, Tesseract ALTO) land here. |
| **MD** | Simple tables + graphics regions + formatted text/headers/lists | ✅ | **The diet — as *annotated* MD.** The vanilla-MD gaps (page borders, bboxes, fonts) are exactly what #10's HTML-comment cues restore at near-zero token cost. |
| **TXT** | Plain ordered text | ⚠️ | **Fallback only** (`xml_to_md.py --format text`); destroys hierarchy and tables. |

## §2 — What existing DU benchmarks actually use (as of 2026-07)

The benchmark landscape splits by task, and its format choices corroborate the two-track design:

- **End-to-end PDF parsing** — **OmniDocBench** (OpenDataLab; already the reference point in
  [`10.plan.md`](10.plan.md) §2b) scores PDF→**Markdown** conversion against MD + layout/bbox
  annotations. Marker/MinerU/Docling all report against it. ⇒ parser quality is *measured in MD*.
- **Table structure** — **PubTabNet / FinTabNet** ground truth is **HTML table fragments**, scored
  by **TEDS** — which [`eval_metrics.py`](../../eval_metrics.py) already implements (optional dep).
  ⇒ tables are the one place HTML survives, as a *scoring* representation, not a diet.
- **Document QA** — DocVQA / MP-DocVQA / DUDE feed **page images** to VLMs; text-side baselines use
  OCR tokens+boxes. ⇒ relevant to #22's VLM arm, not to the text-LLM diet.
- **KIE on scans** — FUNSD / XFUND / CORD / SROIE use **JSON** entity annotations over images.
  ⇒ matches our structured-output side (pydantic schemas), not the input side.
- **OCR/HTR** — ICDAR/READ lines use **PAGE XML**; OCR engines emit PAGE/ALTO. ⇒ our OCR path's
  native landing zone (→ TEITOK), as already planned in #10 §4.
- **Scholarly parsing** — **Grobid** evaluates against **TEI XML** on scientific corpora (header /
  reference extraction). See §4.

**Takeaway:** no benchmark feeds raw HTML+CSS or IDML to a text LLM; parser benchmarks converge on
**MD (+ bbox sidecars)**, recognition on **PAGE/ALTO**, tables on **HTML-via-TEDS**. Our two-track
architecture (annotated-MD diet · ALTO/TEITOK provenance · TEDS available for tables) matches
benchmark practice without inventing anything new.

## §3 — The decision (drafted)

1. **LLM diet = annotated Markdown** (house form): `# <doc-id>` + `## Page N` sections +
   `layout_md.py` HTML-comment cues (`PAGE_BREAK`, `DOC_META`, `BBOX`, `FONT`, `STYLE`,
   `HEADER_*`/`FOOTER_*`, tables as GFM, footnotes as `[^n]`). Produced by
   `api_util/doc_to_visual_md.py` (PDF/DOCX, landed) and `api_util/xml_to_md.py` (TEITOK/ALTO,
   pre-existing; cue-enrichment is a candidate follow-up). One format for both the document-level
   entry point today and the DR's roadmap/agentic navigation later.
2. **Spatial source of truth = PAGE/ALTO/TEITOK** for anything OCR'd or line-level: keeps word/line
   bboxes and `quality_score`/`categ` enrichment; MD is *derived from* it, never the other way.
3. **Page-level citations**: the doc-level prompt already anchors locators to `## Page N`; extend to
   an explicit `[Source: <doc_id>, Page N]` contract + verifier once the roadmap/agent work starts
   (DR §Spatial Provenance) — not before.
4. **HTML/DOCX/PDF/IDML are inputs or intermediates, never the diet; TXT is the degraded fallback.**
5. **FAIR/roadmap architecture** (`SKILL.md`/`INDEX.md` hierarchical navigation, PID/ontology
   alignment) — noted as the DR's target architecture; **deferred** until the #22 benchmark proves
   the basic MD pipeline on in-domain scans. Do not build the index layer before the diet is proven.

## §4 — Grobid and page-classification routing (the issue's two leads)

- **Grobid** (Apache-2.0): mature scholarly-PDF extractor → **TEI XML** (headers, affiliations,
  references, citation contexts). Strong where inputs are born-digital *scientific articles*; weak
  fit for scan-heavy Czech archaeological reports (its models target article layouts, and TEI would
  add a third target format). **Verdict:** do not adopt as the general route; include as a §5
  candidate for the *digital-born scholarly* slice only, and mine its TEI→MD mapping if reference
  extraction ever becomes a requirement.
- **`atrium-page-classification`** routing: use predicted page categories to steer *conversion*, not
  format: e.g. table-heavy pages → table-capable parser path; image/plate pages → `NEEDS_OCR`/VLM;
  text pages → fast extract. Since #10's cues already carry alignment/margins/bboxes inside MD, a
  separate per-page **HTML+CSS branch is unnecessary** — classification decides *treatment*, the
  output stays annotated MD. Evaluate as part of the PoC.

## §5 — PoC + bake-off design (the empirical check)

1. **Sample**: `sample_stratify.py` quality-stratified pages (80/10/10) from the in-domain corpus,
   plus the hardest layouts (multi-column, tables, plates).
2. **Convert** each sample document with the candidates: `doc_to_visual_md.py` (pdfplumber path,
   landed) · **Docling** (MIT; blocked in this sandbox by HF-download policy — run where HF is
   reachable) · **Marker** (license-flagged: GPL + model revenue cap — evaluation only) ·
   `markitdown` baseline · **Grobid** on the digital-born scholarly slice.
3. **Score** with `eval_metrics.py` (CER/WER, normalized edit distance, entity-F1; **TEDS** for
   tables) via `bench_compare.py` against the #22 gold set — plus a **token-cost column**
   (tokens/page per format: annotated-MD vs HTML vs TXT) to quantify the §1 economics claim.
4. **Decide per quality tier** (per #22's guidance): parser choice per page class; confirm or refute
   the `<!-- Page N -->`-marker fidelity through the whole loop (convert → prompt → locator cites
   the right page).
5. **Only then** revisit the FAIR roadmap layer (§3.5).
