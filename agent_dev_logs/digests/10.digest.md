## 🎯 Goal
Decide how to turn input **PDFs (curve-only / scanned / digital-born)** and **DOCX** into the repo's
existing LLM-friendly targets (line-level TEITOK/CSV or document-level Markdown), and pick the
open-source tools to do it.

## 🧭 The answer in one picture
- **Three PDF classes → two paths.** A cheap probe (`pdffonts`/PyMuPDF **font + extractable-char +
  image** census) splits **digital-born** (extract the text layer) from **curve-only + scanned**
  (render @300dpi → OCR). Curves and scans are the *same* OCR path — smoke-tested: `curves.pdf`
  extracted **0** characters, Tesseract `ces` recovered the full text.
- **Watch the 4th class: "extracts but garbled."** Born-digital PDFs with no `/ToUnicode` map decode
  Czech diacritics wrong (`sondě→sondI`) — observed across *every* text parser in the smoke test. Add
  a **decode-sanity check** and send failures to the OCR path.
- **DOCX is the easy input.** Real Unicode layer ⇒ mammoth / pandoc / markitdown / docling all
  round-trip headings + tables near-perfectly. "DOCX/HTML as intermediate" only earns its hop for
  genuinely DOCX-native inputs; prefer **direct PDF→(Markdown or TEITOK)**.

## 🧰 Recommended tools (permissive-first, matches UFAL license pressure)
| Class                  | Primary                                               | Fallback / notes                                                   |
|------------------------|-------------------------------------------------------|--------------------------------------------------------------------|
| Digital-born PDF       | **pdfplumber** (MIT, word-bboxes → TEITOK)            | pymupdf4llm (faster MD, **AGPL**)                                  |
| Curves + scans (OCR)   | **PERO-OCR** (BSD/Apache, **Czech-tuned**, PAGE-XML)  | **Tesseract `ces`** (Apache, emits ALTO → existing `xml_to_md.py`) |
| DOCX / HTML            | **pandoc** (external) or **mammoth** (BSD) → Markdown | Docling/markitdown read DOCX natively                              |
| General complex parser | **Docling** (MIT, CPU-ok, bboxes)                     | **MinerU** for hardest layouts (recheck its custom license)        |

License traps flagged in the plan: **Marker** (GPL + model <$2M revenue cap), **pymupdf4llm/PyMuPDF**
(AGPL), **MinerU** (relicensed off AGPL to a custom Apache-based license), and **flexiconv**
(no declared license — the blocker).

## 🔬 What was actually run (smoke test, CPU-only)
Fabricated a Czech archaeological sample → `digital.pdf` (reportlab), `curves.pdf`
(`gs -dNoOutputFonts`, reproduces issue bullet 1), `scan.pdf` (rasterized, bullet 2), `sample.docx`.
Ran 8 tools. Highlights: routing census discriminates all three classes; pdfplumber gives word
bboxes + table rows; Tesseract `ces` recovers curve-only and scanned text; DOCX conversion flawless
across four tools; **docling PDF path blocked in-session** because it downloads models from
`huggingface.co` (denied by the environment network policy — works where HF is reachable). Full
table in plan §5.
