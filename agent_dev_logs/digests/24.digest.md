## 📦 What's live now (`K4TEL/atrium-llm-enrich@test`, main == test)
- **Engine** (`llm_utils.py`, `vocab_manager.py`, `atrium_paradata.py`, `para_licenses.py`,
  `api_util/{teitok_read,teitok_alto,flexiconv_convert,bbox_scale}.py`) copied byte-identical
  from `atrium-nlp-enrich`, confirmed via diff — no drift.
- **New work** (the point of #24): `openrouter_client.py` (data-sovereignty provider routing,
  optional file-attachment), `ollama_client.py` (native `format` JSON schema, model-pull
  handling), `api_util/xml_to_md.py` (TEITOK/ALTO → Markdown for document-level input), and
  `llm_client_shared.py` — a hand-synced front-end so the remote/lightweight-local backends
  share the quality filter, context-window builder, and schema validation with `llm_utils.py`
  without importing torch.
- **Fixed post-bootstrap**: the initial `Dockerfile` was an untouched copy of `nlp-enrich`'s —
  every build target was broken (`COPY`'d a nonexistent `requirements-test.txt`, `chmod +x`'d
  four `.sh` scripts that were never copied, defaulted `ENTRYPOINT` to a dropped
  `run_pipeline.py`). Rewritten with the repo's actual `base → remote → llm` stages.
  `atrium_paradata.py`'s `_REPO_URLS` fallback map was also missing this repo's own entry.
- **README.md** was the internal strategy doc (`plan.md`) committed verbatim — replaced with
  real usage docs (setup, all four backends, model registry, Docker).
- **`tests/`** — new: `test_llm_client_shared.py` (22 tests) and `test_xml_to_md.py` (15 tests),
  targeting exactly the two genuinely-new modules; the shared-engine files are covered by
  `atrium-nlp-enrich`'s own suite and intentionally not re-tested here.
- **`.github/workflows/`** — full 8-workflow suite (`docker`, `pre-commit`, `codeql`,
  `security`, `scheduled-smoke`, `gpu-inference`, `para-drift`, `release`) + `dependabot.yml`,
  adapted from `atrium-nlp-enrich`'s real configs. One caught-before-ship issue: the
  docker-tool reusable maps `build-targets: "base"` to Docker's *last declared stage*, not
  literally `base` — this repo's `build-targets` is `["remote", "llm"]` only.
- **`docker-compose.yaml` / `docker-compose.gpu.yaml`** — two profile-gated services
  (`llm-enrich-remote`, `llm-enrich-llm`); no unprofiled default, since neither is safe to run
  without explicit model/API-key config.
- **`CONTRIBUTING.md`** — adapted from the hub template + `nlp-enrich`'s real example.
