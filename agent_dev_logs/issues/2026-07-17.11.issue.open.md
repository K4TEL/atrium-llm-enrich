### 1. Why Markdown Wins (Token Economics vs. Structure)

- **The HTML Problem:** While HTML accurately preserves layout, it causes severe token bloat. Tags like `<div>`, `<span>`, and inline styles rapidly exhaust the LLM context window without adding semantic value.

- **The Plain Text Problem:** Stripping everything to raw text destroys hierarchical relationships, reading order (especially in multi-column PDFs), and tabular data.

- **The Markdown Sweet Spot:** Markdown strikes the optimal mathematical balance. It preserves structural semantics (headers, lists, emphasis) and tabular data using a highly token-efficient syntax that native LLM self-attention mechanisms are already heavily optimized to understand.

### 2. Handling Complex Layouts and Tables

Legacy parsers (like PyMuPDF) will fail on multi-column layouts and complex tables. We should look into deploying advanced vision-language parsers like Docling (IBM) or Marker.

- **Tables:** Docling, for instance, uses an Optimized Table Structure Language (OTSL) tokenization scheme that represents complex, merged-cell tables using about 80% fewer tokens than HTML, while maintaining high accuracy.

- **Math & Scientific Notation:** The parser must be configured to convert inline and block equations into standard LaTeX syntax (e.g., `$E=mc^2$ or $$...$$`).

### 3. CRITICAL: Injecting Spatial Provenance (Page Markers)

Because one of our core requirements is returning page-level citations, standard Markdown isn't enough (it acts as an infinite scroll). We need to explicitly inject bounding markers into the Markdown stream during the parsing phase.

As the parser iterates through the document, it should extract the native page label from the PDF metadata (e.g., `"iv"` or `"14"`) and insert an HTML comment right before that page's text begins:

```bash
<!-- Page 14 -->
The text from page 14 begins here...

```

By embedding these markers and explicitly prompting the LLM to use them, the model internalizes the spatial boundaries of the document. We can then enforce a rigid citation format in the system prompt (e.g., `[Source: <doc_id>, Page: 14]`), which allows our verifier agent to actually check the claims against the physical document.

### Next Steps / Recommendation:

I recommend we run a PoC using Docling or Marker to parse a sample of our most complex PDFs into Markdown, explicitly testing the injection of `<!-- Page N -->` markers and LaTeX equation formatting.

Based on recent architectural research into Document Understanding (DU) and FAIR data navigation, here is a summary of findings to guide our decision on LLM input formats (By Gemini 3.1 Pro Deep Research [research-issue-11.md](https://github.com/user-attachments/files/30216203/research-issue-11.md)):

### 1. Markdown is the Optimal Lingua Franca
The consensus in modern agentic architectures points heavily to **Markdown (MD)** as the optimal input format for LLMs. 
* **Why not HTML/XML?** While HTML accurately preserves layout, it suffers from severe token bloat due to excessive structural tags (like `<div>` or `<span>`), rapidly exhausting the LLM's context window.
* **Why not TXT?** Plain text strips away critical hierarchical and tabular information, destroying the logical relationships between data points.
* **The MD Advantage:** Markdown preserves structural semantics (headers, lists, tables) using a highly token-efficient syntax that native LLM self-attention mechanisms process highly effectively.

### 2. Solving Spatial Provenance (Page-Level Citations)
A major advantage of formats like Page/ALTO XML is spatial positioning. To replicate this in Markdown and enable precise page-level citations, modern ingestion pipelines explicitly inject **bounding markers**.
* As the document is parsed, markers like `<!-- Page N -->` or `[Page N]: #` are explicitly embedded into the Markdown stream at physical page boundaries.
* This allows the LLM to maintain a continuous spatial map of the document and append verifiable reference links to its generated output.

### 3. Recommended Ingestion Tooling
While Grobid is a solid benchmark tool, research highlights newer advanced vision-language models and layout analysis toolkits like **Docling** (from IBM Research) and **Marker**.
* These tools perform comprehensive layout analysis and Table Structure Recognition (TSR) to correctly reconstruct logical reading orders and convert complex tables into token-efficient Markdown without legacy heuristic failures.
* Docling natively supports extracting PDF page labels to inject the necessary bounding markers.

### 4. FAIR Alignment via Hierarchical Metadata
To ensure FAIR compliance, the raw Markdown documents shouldn't be processed as isolated chunks. Instead, they should be accompanied by explicitly extracted metadata (topics, Named Entities mapped to standardized ontologies). Structuring this metadata in a hierarchical "roadmap" (e.g., using `SKILL.md` and `INDEX.md` files) allows the LLM agent to deterministically navigate the corpus before extracting the full-length MD files for synthesis.

**Conclusion:** We should prioritize a pipeline that converts PDFs and source documents into **annotated Markdown** (with embedded page markers), utilizing tools like Docling or Marker to ensure high structural fidelity, token efficiency, and verifiable provenance.

**Decision (drafted) + mechanism partly built.**

- **Diet = annotated Markdown.** MD is the token-efficiency/structure sweet spot; the gaps vs PAGE/ALTO (page borders, positions, fonts) are restored by HTML-comment cues at near-zero token cost. **HTML** stays an intermediate only (token bloat), **IDML** is skipped, **PAGE/ALTO/TEITOK** is the spatial source of truth, **TXT** is the degraded fallback.
- **Grounded in DU benchmarks:** end-to-end PDF parsing is scored *in Markdown* (OmniDocBench); table structure via **TEDS** (already in `eval_metrics.py`); recognition ground truth is **PAGE/ALTO**. No benchmark feeds raw HTML+CSS/IDML to a text LLM — so our two-track design matches practice.
- **Grobid / page-classification:** Grobid (→ TEI) fits *born-digital scholarly* PDFs, not scan-heavy reports — keep as a bake-off candidate for that slice only; page-classification should steer *treatment* (extract vs OCR vs table-parser), output stays annotated MD.
- **Already built (see #10):** all three source families now emit the annotated MD (`doc_to_visual_md` + `xml_to_md --format layout`), and the doc-level schema carries a `page` field for citations.

**Next (empirical):** Docling/Marker PoC on the hardest PDFs (incl. `<!-- Page N -->` round-trip + LaTeX), then a format bake-off through the #22 harness (`sample_stratify` → converters → `eval_metrics`/TEDS + a token-cost column) to decide per quality tier.
