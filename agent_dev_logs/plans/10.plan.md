## Context
Issue #10 asks how to turn **input PDFs of several kinds** — and **DOCX** as an alternative — into
**LLM-friendly inputs** (text + whatever layout signal we can keep). The issue names three PDF
classes plus two format asks:

1. **Curves instead of letters** — text drawn as vector outlines; no text layer, no fonts to map
   back to Unicode.
2. **Scanned images** — image-only pages, no text layer.
3. **Digital-born with text** — an extractable text layer already present.
4. **DOCX** as an alternative input format to PDF.
5. **DOCX / HTML as intermediate formats** on the PDF→LLM path (the issue's own suggestion).

**What "LLM-friendly" means in this repo.** Two existing entry points already define the target
shapes, so a converter should aim at one of them rather than invent a third:
- **Line-level** — `INPUT_DIR/*.csv` or `*.teitok.xml` (`file_id`/`page_num`/`line_num`/`categ`/
  `quality_score`/`text`), consumed by `llm_run.py` (local) and the `run_line_level()` path in
  `llm_client_shared.py`.
- **Document-level** — `.md`/`.txt`, consumed by `run_document_level()` in the remote/Ollama
  clients, today produced by [`api_util/xml_to_md.py`](../../api_util/xml_to_md.py) from TEITOK/ALTO.

So the pipeline already eats **TEITOK XML**, **CSV**, and **Markdown/plain text**. A PDF/DOCX
front-end's job is to land in one of those — most naturally **TEITOK** (line-level, keeps bboxes)
or **Markdown** (document-level).

**What already exists (and its limits).**
- [`api_util/flexiconv_convert.py`](../../api_util/flexiconv_convert.py) wraps UFAL's own
  [`flexiconv`](https://github.com/ufal/flexiconv) (pdf/docx/odt/rtf/html/md/txt/epub/… → TEITOK
  XML). It is a **standalone helper — nothing in the run pipeline imports it**, and
  [`para_config.txt:32-33`](../../para_config.txt) gates it: *"license not yet checked against
  project policy; flag before enabling txt/pdf/docx conversion."* (Confirmed: the public flexiconv
  repo declares **no license** in its README/repo metadata — the gate is real.)
- [`api_util/xml_to_md.py`](../../api_util/xml_to_md.py) is XML→text only (TEITOK/ALTO → page-sectioned
  Markdown). It is the house style for "→ LLM-friendly text" and the model a new converter should match.

**Relation to hub [#22](https://github.com/ufal/atrium-project/issues/22) (Document Understanding).**
#22 is the DU **evaluation** track (VLM-vs-legacy benchmark on in-domain Czech archaeological scans;
`bench_compare.py` + `sample_stratify.py` + `eval_metrics.py`). Its explicit guidance:
**do not build the ALTO→MD→LLM→TEITOK "Markdown-bridge" until the benchmark proves it is worth it**,
and it already noted **`opendatalab/MinerU`** as the parser candidate for that route. This issue (#10)
is the **input-ingestion** counterpart: the tools below are exactly the candidates #22 would
benchmark, so pick the routing here and let #22's metrics choose the winner per quality tier.

## §1 — Input taxonomy & routing

The three PDF classes collapse to **two processing paths**, decided by a cheap, dependency-light
probe. A digital-born page has an extractable text layer and named fonts; curve-only and scanned
pages have neither, so both fall through to render-then-OCR.

**Detection heuristic (per page, then aggregate):**
- `pdffonts` (poppler) or `PyMuPDF page.get_fonts()` → **font count**.
- `PyMuPDF page.get_text("text")` → **extractable-char count**.
- `page.get_images()` → **image count** (a page that is one big image ⇒ scan).
- **Route:** `chars > threshold and fonts > 0` → **DIGITAL-BORN** (extract text layer);
  else → **RENDER + OCR** (covers curves *and* scans). A **per-page** decision matters: real reports
  mix a born-digital cover with scanned plates.

**Fourth, sneaky class — "extracts but garbled".** A digital-born PDF whose fonts are subsetted with
**no `/ToUnicode` CMap** will *extract* characters that are the wrong Unicode (diacritics especially).
This is not hypothetical — it showed up in our own smoke fixture (§5): text came out with `ě→I`,
`ř→I`, `ů→I`. Guard against it with a **decode-sanity check** (ratio of `U+FFFD`/control/out-of-script
codepoints, or a language-model/dictionary hit-rate on the extracted text); when it fails, treat the
page as OCR-path even though `chars > 0`. This is the single most important detail for Czech text.

Smoke-tested discrimination (fabricated fixtures, §5):

| fixture       | fonts | text chars | images | routed to              |
|---------------|------:|-----------:|-------:|------------------------|
| `digital.pdf` |     3 |        387 |      0 | DIGITAL-BORN (extract) |
| `curves.pdf`  |     0 |          0 |      0 | RENDER + OCR           |
| `scan.pdf`    |     0 |          0 |      1 | RENDER + OCR           |

**Takeaway for #4/#5:** DOCX is the *easy* input — it has a real Unicode text layer, so mammoth/
pandoc/markitdown/docling all round-trip it near-perfectly (§5). "DOCX/HTML as **intermediate**"
(the issue's #5) is only worth it on the PDF side when a tool reconstructs structure better via DOCX
than via direct MD; in practice prefer **direct PDF→(MD or TEITOK)** and reserve DOCX/HTML
intermediates for genuinely DOCX-native inputs. Avoid `pdf2docx` (MIT) as a bridge — it is lossy and
adds a hop.

## §3 — Target format for the LLM

- **Markdown** — best token-efficiency/structure trade-off; what most parsers emit; lands directly on
  the **document-level** entry point. Use for whole-document enrichment.
- **TEITOK XML** — the repo's **line-level** native; keeps `page/line/bbox` so per-line `categ`/
  `quality_score` enrichment and paradata still work. Prefer when we need line granularity or bbox
  provenance (OCR path via PERO PAGE-XML / Tesseract ALTO → TEITOK).
- **HTML** — a faithful *intermediate* (mammoth/LibreOffice), not the LLM's final diet; convert on to
  MD/TEITOK.
- **DocTags** (SmolDocling/Docling) — richest structure, but couples us to Docling; fine internally,
  convert to MD/TEITOK at the boundary.

Recommendation: **do not add a new target format.** Route digital-born + OCR output into **ALTO/
PAGE-XML → TEITOK** (reusing `xml_to_md.py` and the TEITOK utilities) for line-level, or into
**Markdown** for document-level. This is exactly the seam `xml_to_md.py` already occupies.

## §4 — Recommended routing for `atrium-llm-enrich` (no code landed here)

A thin **router** in front of the existing dispatch, mirroring `flexiconv_convert.py`'s standalone
shape but producing the repo's native targets:

```
input file
  ├─ .csv / .teitok.xml / .md / .txt   → (unchanged) existing dispatch
  ├─ .docx / .html / .odt / .rtf       → pandoc|mammoth → Markdown (doc-level)   [+ Docling if adopted]
  └─ .pdf → probe (§1)
        ├─ DIGITAL-BORN  → pymupdf4llm/pdfplumber → Markdown/TEITOK
        │      └─ decode-sanity fails → treat as OCR
        └─ CURVES | SCAN → render @300dpi → OCR (PERO-OCR PAGE-XML | Tesseract `ces` ALTO)
                              → TEITOK (line-level) or Markdown (doc-level)
```

Integration seams (for the follow-up issue, not this one): the extension check in
[`llm_run.py:413`](../../llm_run.py) and the dispatch in
[`llm_client_shared.py`](../../llm_client_shared.py) (`read_input_rows` / `run_line_level` /
`run_document_level`). A converter should emit `.teitok.xml` or `.md` into `INPUT_DIR` so **no
dispatch change is needed** — pre-convert, then run, exactly as `flexiconv_convert.py` is meant to be
used.

**Tooling recommendation (permissive-first, matches UFAL policy pressure in `para_config.txt`):**
- **Digital-born PDFs:** `pdfplumber` (MIT, word-bboxes → TEITOK) or `pymupdf4llm` (faster MD, but
  **AGPL** — prefer pdfplumber for a shared service unless we license PyMuPDF).
- **Curves + scans:** **PERO-OCR** (BSD/Apache, Czech-tuned, PAGE-XML) as the primary; **Tesseract
  `ces`** (Apache) as the zero-GPU fallback that already emits ALTO for `xml_to_md.py`.
- **DOCX/HTML:** **pandoc** (GPL binary, used as an external tool — not linked) or **mammoth** (BSD)
  → Markdown.
- **General/complex parser (optional, evaluate in #22):** **Docling** (MIT) for the best
  structure-preserving MD/JSON when CPU-only permissive parsing is wanted; **MinerU** for the hardest
  layouts (recheck its custom license).
- **flexiconv:** keep as the one-shot convenience wrapper **only after its license is resolved**
  (blocker above); it already covers this whole matrix in one call, which is why resolving its
  license is the highest-leverage unblock.

Deliberately **not** doing now (per hub #22 risk note): building the full deterministic
ALTO→MD→LLM→TEITOK routing bridge. Land the router + one tool per class behind a benchmark first.

**Two findings worth carrying forward:**
1. **OCR is mandatory for class 1 (curves).** `curves.pdf` extracted **0** characters via every text
   parser; Tesseract `ces` recovered the whole paragraph. Curves and scans are the *same* path.
2. **Diacritic corruption on digital-born is real.** `digital.pdf` (non-embedded WinAnsi Helvetica,
   no `/ToUnicode`) extracted `sondě→sondI`, `hřeby→hIeby` across *all* text parsers — a systematic,
   not random, error. Hence the §1 **decode-sanity check**: born-digital ≠ trustworthy text for Czech.
