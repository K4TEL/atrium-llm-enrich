### **Exhaustive Taxonomy of Visual Layout Insertions**

To fully leverage the capabilities of advanced PDF-to-MD and DOCX-to-MD tools, a comprehensive and standardized schema for translating visual layouts into Markdown and HTML structures is required. The following taxonomy details an exhaustive list of visual layout pieces, their optimal encoding strategies, and the specific causal advantages they provide to LLM processing.

#### **Macro-Structural Boundaries and Page Borders**

The most fundamental visual cues in a physical document dictate the macro-structure of the page. Understanding page boundaries, canvas dimensions, and page borders is essential for LLMs when performing document Question Answering (QA), as it allows the model to accurately cite the location of extracted information and understand the physical flow of a narrative across printed media.  
Page borders and margins are particularly critical. In physical documents, the visual framing and physical white space surrounding the core text block separate the main narrative from marginalia, headers, footers, and stamps. If margin data is not encoded, an LLM might erroneously merge a margin note into the middle of a continuous paragraph, destroying the semantic integrity of the text.

| Visual Layout Element | Description and Purpose | Optimal MD / HTML Encoding Strategy | LLM Processing and Reasoning Advantage |
| :---- | :---- | :---- | :---- |
| **Page Break** | The physical termination of one page and the beginning of the next. | \<\!-- PAGE\_BREAK: pg\_12 \--\> \--- | Provides a strict contextual boundary, anchoring the LLM's internal mapping of document length and enabling precise citation generation. |
| **Page Dimensions** | The absolute width and height of the physical page canvas. | \<\!-- DOC\_META: size=8.5x11in, orientation=portrait \--\> | Informs the LLM of the canvas shape, providing a mathematical baseline for interpreting all subsequent relative bounding box coordinates. |
| **Page Borders and Margins** | The defined white space or physical lines separating core text from the page edge. | \<\!-- LAYOUT\_MARGIN: top=1in, bottom=1in, left=1in, right=1in \--\> | Allows the model to explicitly distinguish between core body text and marginalia, preventing the semantic merging of unrelated text streams. |
| **Multi-Column Formatting** | The division of text into vertical columns, standard in scientific papers and newsprint. | \<\!-- LAYOUT\_COLUMN: 1\_of\_2 \--\> text \<\!-- LAYOUT\_COLUMN: 2\_of\_2 \--\> | Preserves the intended reading order while explicitly defining that the text blocks were physically separated on the page canvas. |

#### **Spatial Geometry and Coordinate Data**

For tasks requiring high spatial precision, such as directing an autonomous agent to interact with a specific graphical element or extracting data based on physical proximity, bounding boxes must be recorded. To avoid the token bloat associated with line-by-line coordinate injection, these coordinates should be aggregated at the highest logical level, typically the paragraph or discrete text block.

| Visual Layout Element | Description and Purpose | Optimal MD / HTML Encoding Strategy | LLM Processing and Reasoning Advantage |
| :---- | :---- | :---- | :---- |
| **Block Bounding Box** | The strict \[x\_min, y\_min, x\_max, y\_max\] coordinates of a cohesive paragraph or image. | \<\!-- BBOX: \[120, 400, 600, 550\] \--\> | Emulates the Layout-as-Token architecture, allowing the LLM to deduce physical proximity, calculate overlap, and infer relationships between adjacent text blocks. |
| **Text Alignment** | Left, right, centered, or justified physical text positioning within a block. | \<div align="center"\> \#\# Title \</div\> | Centers often indicate primary headings or pull quotes; HTML alignment tags are natively understood by LLMs as markers of structural hierarchy. |
| **Indentation and Tabulation** | Physical spacing applied at the beginning of a line or paragraph. |      or \<\!-- INDENT: 4px \--\> | Indicates hierarchical nesting, which is particularly vital for the LLM when parsing poorly formatted lists, legal sub-clauses, or code blocks extracted from PDFs. |
| **Line Spacing (Leading)** | The vertical distance between baselines of successive lines of type (e.g., single, double). | \<\!-- STYLE: line-spacing=1.5 \--\> | Frequently utilized in legal and historical documents to denote specific clauses, drafts, or later additions; capturing this preserves critical legal semantic context. |

#### **Typographical and Stylistic Semantics**

While basic Markdown inherently supports elementary typography—specifically bolding and italicization—DOCX and PDF files contain highly nuanced font data that can carry immense semantic meaning. For example, in complex legal contracts, a specific font family or capitalization scheme may be used exclusively for defining strict legal terms. If this typographical nuance is flattened, the LLM loses the ability to distinguish a globally defined term from standard prose1.

| Visual Layout Element | Description and Purpose | Optimal MD / HTML Encoding Strategy | LLM Processing and Reasoning Advantage |
| :---- | :---- | :---- | :---- |
| **Basic Emphasis** | Bold and Italicized text indicating stress or titles. | \*\*bold\*\* or \*italic\* | Natively parsed by the LLM as markers of high importance, subtly altering the probability distribution of attention towards these highlighted entities12. |
| **Font Size and Weight** | The exact point size and numerical weight (e.g., 14pt, 700 weight). | \<\!-- FONT: size=14pt, weight=700 \--\> | Allows the LLM to dynamically reconstruct the heading hierarchy and infer importance, even if standard Markdown hashes failed to generate during extraction. |
| **Font Family / Typeface** | The specific typeface category used (e.g., Courier, Times New Roman, Arial). | \<\!-- FONT: family="Courier New" \--\> | Typeface changes reliably signal a shift in content type, such as monospace fonts indicating embedded code snippets, mathematical formulas, or financial figures. |
| **Text Color and Highlighting** | The specific hex color of the font or background highlight. | \<\!-- STYLE: color=\#FF0000, highlight=yellow \--\> | Captures human annotations, critical warnings, or redactions that would otherwise be permanently lost in a standard plaintext conversion pipeline. |
| **Strikethrough and Underline** | Text that is physically crossed out or underscored with a line. | \~\~deleted text\~\~ or \<u\>underlined\</u\> | Absolutely essential for tracking version changes in legal documents, collaborative drafts, and historical archive analysis. |

#### **Document-Specific Embedded Structures**

PDFs and Word documents contain specialized, highly structured elements that organize data visually. The translation of these elements into Markdown requires specific syntactic treatments to ensure the LLM understands their distinct nature separate from the main narrative flow.

| Visual Layout Element | Description and Purpose | Optimal MD / HTML Encoding Strategy | LLM Processing and Reasoning Advantage |
| :---- | :---- | :---- | :---- |
| **Headers and Footers** | Repeating text blocks located at the top or bottom margin of every page. | \<\!-- HEADER\_START \--\> text \<\!-- HEADER\_END \--\> | Prevents the LLM from hallucinating by confusing repeating document titles, dates, or page numbers with the main narrative body text. |
| **Footnotes and Endnotes** | Super-scripted numerical references linking to expanded text at the bottom of the page or document. | Text\[^1\] \[^1\]: Note content. | Standard Markdown footnote syntax perfectly maps to DOCX footnotes, preserving complex academic and legal citation chains natively for the LLM to traverse. |
| **Tabular Data** | Data organized explicitly in intersecting rows and columns. | Standard MD Tables: | Col A | Col B | | Presents tabular information in a highly structured format that the LLM can reason about directly, preserving row/column relationships without inducing spatial hallucinations8. |
| **Embedded Images and Figures** | Raster or vector graphics placed within the text flow, often containing charts or photographs. | \!\[Alt text\](image\_link) \<\!-- BBOX: \[x,y,w,h\] \--\> | Provides a localized placeholder for multi-modal Vision Language Models (VLMs) to cross-reference the surrounding text with the precise visual data located at those coordinates. |
| **Watermarks** | Faded, often diagonal text overlaid across the entire page canvas (e.g., "CONFIDENTIAL", "DRAFT"). | \<\!-- WATERMARK: "CONFIDENTIAL DRAFT" \--\> | Informs the LLM of the document's global state and security level, which is highly relevant for automated document classification and compliance tasks. |

### Knowledge Base Architecture: Vector Embeddings vs. Flat Markdown

* **Flat Markdown as Source of Truth:** Acts as a portable, vendor-agnostic foundation that remains readable and independent of proprietary AI tools.
* **Vector Embeddings for Scale:** Essential for massive enterprise archives to enable semantic search, multi-modal retrieval, and concept discovery beyond context window limits.
* **Optimal Hybrid Architecture:** Treat visually-enriched flat Markdown as the immutable primary source, while deriving, calculating, and caching advanced vector layouts and semantic indices from it.

### Automating Visual Extraction and Structuring

To successfully feed downstream data pipelines, LLM outputs must be strictly constrained using three core methods:

* **Format Prompting:** Force specific output strings (e.g., strict CSV with zero conversational filler).
* **Self-Parsing JSON:** Instruct the model to map responses directly to predefined, rigid JSON schemas.
* **Post-Processing Scripts:** Use lightweight scripts to strip conversational wrappers and validate extracted coordinate bounds against original page dimensions.

### Synthesis and Strategic Outlook

* **Hybrid Encoding Strategy:** Combine the minimalist syntax of Markdown for human-readable structures with invisible HTML comments for machine-readable spatial metadata.
* **Efficiency Gains:** Prevents token bloat and bypasses the severe sequence-length issues of continuous coordinate-as-tokens schemes.
* **Document Understanding:** Converts physical layouts into logical, markdown-driven semantics, allowing AI to comprehend complex documents as interconnected architectures of human information.

**Update — implemented on `test` (visually-rich Markdown pipeline).**

- **Converter landed:** `api_util/doc_to_visual_md.py` (+ `docx_to_md.py`, `pdf_to_md.py`) turns **DOCX** and **digital-born PDF** into page-sectioned Markdown, recording page borders and layout cues as HTML comments (`<!-- PAGE_BREAK -->`, `<!-- DOC_META -->`, `<!-- BBOX -->`, `<!-- FONT -->`, `<!-- STYLE -->`, headers/footers, tables, footnotes). The full taxonomy from the comment above is single-sourced in `api_util/layout_md.py` (`CUE_SCHEMA`).
- **Unified schema:** `api_util/xml_to_md.py --format layout` now emits the *same* cues for **TEITOK/ALTO** from their coordinates — so PDF, DOCX and ALTO/TEITOK all produce one annotated-Markdown format (see #11).
- **OCR path (opt-in):** scanned / curve-only pages are flagged `<!-- NEEDS_OCR -->` by default; with `--ocr` they're rendered (pypdfium2) and transcribed with **Tesseract `ces`**, tagged `<!-- OCR: engine=tesseract, lang=ces -->`. Final OCR-tool choice (PERO-OCR vs Tesseract) stays gated on the #22 benchmark.
- **Pipeline integration:** the openrouter/ollama clients **auto-convert** `.pdf`/`.docx` dropped in `INPUT_DIR` (cached `.md`), and the document-level schema returns a `page` field for `[Source: <doc_id>, Page N]` citations. No dispatch change; HTML-comment cues pass through as inert text.

Permissive-first deps (`python-docx`, `pdfplumber`, `pytesseract` — all MIT/Apache) in `requirements_docmd.txt`. Full test suite green. **Still open:** flexiconv's undeclared upstream license (`para_config.txt`) remains a blocker for that one-shot path.
